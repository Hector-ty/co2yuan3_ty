# 用户权限管理问题评估报告

## 执行时间
2024年（当前时间）

## 评估范围
- 后端权限验证机制
- 前端权限控制
- 数据访问控制
- 角色定义和权限分配
- 安全漏洞

## 一、发现的主要问题

### 🔴 严重问题

#### 1. **硬编码的root用户凭据**
**位置**: `server/controllers/auth.js:18`
**问题描述**:
- root用户的邮箱和密码硬编码在代码中（`root@root.com` / `root1234`）
- 这是一个严重的安全隐患，任何能访问代码的人都能获得超级管理员权限
- 密码强度极低，容易被暴力破解

**影响**:
- 高风险：可能导致未授权访问
- 违反安全最佳实践

**建议修复**:
```javascript
// 应该使用环境变量
const ROOT_EMAIL = process.env.ROOT_EMAIL || 'root@root.com';
const ROOT_PASSWORD = process.env.ROOT_PASSWORD; // 必须从环境变量读取
```

---

#### 2. **公开的数据访问端点**
**位置**: `server/routes/carbonData.js:12`
**问题描述**:
- `GET /api/carbon-data` 使用 `optionalAuth` 中间件
- 未认证用户可以访问所有碳排放数据
- 虽然控制器中有权限检查，但未认证时可能返回所有数据

**代码片段**:
```12:12:server/routes/carbonData.js
router.get('/', optionalAuth, getData);
```

**影响**:
- 高风险：敏感数据可能被未授权访问
- 违反数据隐私保护原则

**建议修复**:
- 如果移动端需要公开访问，应该创建独立的路由（如 `/api/mobile/carbon-data`）
- 主API应该要求认证

---

#### 3. **权限检查机制不统一**
**问题描述**:
- 部分路由使用 `checkPermission`（基于权限配置）
- 部分路由使用 `authorize`（基于角色列表）
- 用户管理API（`/api/auth/users`）在控制器中手动检查角色，而不是使用中间件

**位置**: 
- `server/routes/auth.js:11-14` - 只使用 `protect`，权限检查在控制器中
- `server/routes/carbonData.js` - 使用 `checkPermission`
- `server/routes/emissionFactors.js` - 使用 `authorize`

**影响**:
- 中风险：权限检查逻辑分散，难以维护
- 容易出现遗漏或不一致

**建议修复**:
- 统一使用 `checkPermission` 中间件
- 将权限检查从控制器移到路由层

---

### 🟡 中等问题

#### 4. **角色迁移逻辑的潜在问题**
**位置**: `server/middleware/permissions.js:14-22`
**问题描述**:
- 在权限检查中间件中进行角色映射（editor → organization_user）
- 这种临时映射可能导致权限判断不准确
- 如果数据库中的角色没有及时迁移，可能出现权限错误

**代码片段**:
```14:22:server/middleware/permissions.js
// 兼容性处理：如果用户是旧角色，临时映射到新角色（实际升级在登录时完成）
let userRole = user.role;
if (userRole === 'editor') {
    userRole = 'organization_user';
} else if (userRole === 'admin') {
    userRole = 'province_admin';
} else if (userRole === 'viewer') {
    userRole = 'organization_user';
}
```

**影响**:
- 中风险：可能导致权限判断不一致
- 维护困难

**建议修复**:
- 在登录时强制迁移所有旧角色
- 数据库迁移脚本确保所有用户角色都是新角色
- 移除中间件中的临时映射逻辑

---

#### 5. **用户管理API缺少统一的权限检查**
**位置**: `server/controllers/auth.js:274-351`
**问题描述**:
- `getAllUsers`, `updateUserRole`, `deleteUser` 在控制器中手动检查角色
- 没有使用统一的权限检查中间件
- 权限检查逻辑重复且容易出错

**代码片段**:
```274:280:server/controllers/auth.js
exports.getAllUsers = async (req, res, next) => {
  try {
    // 只有超级管理员和管理员角色可以访问
    const adminRoles = ['superadmin', 'province_admin', 'city_admin', 'district_admin'];
    if (!adminRoles.includes(req.user.role)) {
      return res.status(403).json({ success: false, error: 'Not authorized to access this route' });
    }
```

**影响**:
- 中风险：权限检查逻辑分散
- 代码重复，维护困难

**建议修复**:
- 在路由层使用 `checkPermission(PERMISSIONS.USERS_READ)` 等中间件
- 移除控制器中的手动检查

---

#### 6. **前端权限控制依赖客户端数据**
**位置**: `client/src/App.jsx:15-31`, `client/src/pages/DashboardPage.jsx:97-103`
**问题描述**:
- 前端权限控制主要依赖 `localStorage` 中的用户信息
- 客户端数据可以被轻易篡改
- 虽然有后端验证，但前端显示可能误导用户

**代码片段**:
```15:31:client/src/App.jsx
const ProtectedRoute = ({ children, requiredRole = null }) => {
    const token = localStorage.getItem('token');
    if (!token) {
        return <Navigate to="/login" />;
    }
    
    if (requiredRole) {
        const user = JSON.parse(localStorage.getItem('user') || '{}');
        // 支持单个角色或角色数组
        const allowedRoles = Array.isArray(requiredRole) ? requiredRole : [requiredRole];
        if (!allowedRoles.includes(user.role)) {
            return <Navigate to="/dashboard" />;
        }
    }
    
    return children;
};
```

**影响**:
- 中风险：用户可能看到不应该看到的功能（虽然无法实际使用）
- 用户体验问题

**建议修复**:
- 前端权限控制仅用于UI显示优化
- 所有实际权限验证必须在后端完成
- 定期从后端验证用户权限状态

---

#### 7. **权限配置与实际使用不一致**
**位置**: `server/config/permissions.js`
**问题描述**:
- 权限配置定义了 `CARBON_DATA_CREATE` 权限
- 但 `organization_user` 角色拥有 `CARBON_DATA_CREATE` 权限
- 而实际代码中，机构用户只能创建自己的数据，管理员可以创建权限范围内的数据
- 权限粒度不够细，无法区分"创建自己的数据"和"创建任意数据"

**影响**:
- 低-中风险：权限模型不够精确
- 未来扩展困难

**建议修复**:
- 考虑更细粒度的权限定义
- 或者使用基于资源的权限检查（RBAC + 资源所有权）

## 二、权限架构分析

### 当前角色体系
1. **superadmin** (超级管理员)
   - 拥有所有权限
   - 可以访问所有数据
   - 可以管理所有用户

2. **province_admin** (省级管理员)
   - 可以管理本省及以下用户
   - 可以查看和更新本省及以下数据
   - 不能管理其他省级管理员

3. **city_admin** (市级管理员)
   - 可以管理本市及以下用户
   - 可以查看和更新本市及以下数据
   - 不能管理省级和市级管理员

4. **district_admin** (区县级管理员)
   - 可以管理本区县用户
   - 可以查看和更新本区县数据
   - 不能管理省级、市级和区县级管理员

5. **organization_user** (机构用户)
   - 只能查看和修改自己的数据
   - 不能管理其他用户

### 权限配置分析
- ✅ 角色定义清晰
- ✅ 权限分配合理
- ⚠️ 但实际使用中，权限检查逻辑分散，不够统一

## 三、数据访问控制分析

### 碳排放数据访问控制
1. **创建数据** (`POST /api/carbon-data`)
   - ✅ 使用 `checkPermission(PERMISSIONS.CARBON_DATA_CREATE)`
   - ✅ 机构用户强制使用自己的地区代码
   - ✅ 超级管理员可以指定任意地区

2. **读取数据** (`GET /api/carbon-data`)
   - ⚠️ 使用 `optionalAuth`，允许未认证访问
   - ✅ 已认证用户根据角色过滤数据
   - ⚠️ 未认证用户可能看到所有数据

3. **更新数据** (`PUT /api/carbon-data/:id`)
   - ✅ 使用 `checkPermission(PERMISSIONS.CARBON_DATA_UPDATE)`
   - ✅ 机构用户只能更新自己的数据
   - ✅ 管理员只能更新权限范围内的数据

4. **删除数据** (`DELETE /api/carbon-data/:id`)
   - ✅ 使用 `checkPermission(PERMISSIONS.CARBON_DATA_DELETE)`
   - ✅ 机构用户只能删除自己的数据
   - ✅ 管理员只能删除权限范围内的数据

### 用户管理访问控制
1. **获取用户列表** (`GET /api/auth/users`)
   - ⚠️ 只在控制器中检查角色，未使用权限中间件
   - ✅ 根据管理员级别过滤可见用户

2. **更新用户角色** (`PUT /api/auth/users/:id/role`)
   - ⚠️ 只在控制器中检查角色
   - ✅ 检查权限范围，防止越权操作

3. **删除用户** (`DELETE /api/auth/users/:id`)
   - ⚠️ 只在控制器中检查角色
   - ✅ 检查权限范围
